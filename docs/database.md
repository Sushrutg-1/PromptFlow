# Database Documentation

This document describes the database design used in **PromptFlow**.

PromptFlow uses **MongoDB** as its primary database and **Mongoose** as the ODM (Object Document Mapper).

The database is designed to provide:

- Scalable conversation management
- Secure user authentication
- Multi-model AI responses
- Easy future expansion

---

# Database Overview

PromptFlow currently consists of four collections.

```text
User
 │
 └──────────────┐
                │
                ▼
        Conversation
                │
                ▼
             Turn
                │
                ▼
            Response
```

---

# Collection Overview

| Collection   | Purpose                                                     |
| ------------ | ----------------------------------------------------------- |
| User         | Stores user account information and authentication details. |
| Conversation | Stores conversation metadata created by a user.             |
| Turn         | Stores each prompt sent within a conversation.              |
| Response     | Stores AI-generated responses for a prompt.                 |

---

# User Collection

Stores account information and authentication data.

## Fields

| Field          | Type     | Description                      |
| -------------- | -------- | -------------------------------- |
| `_id`          | ObjectId | Unique user identifier.          |
| `name`         | String   | User's full name.                |
| `email`        | String   | Unique email address.            |
| `password`     | String   | Hashed password.                 |
| `avatar`       | String   | Profile image URL.               |
| `role`         | String   | User role.                       |
| `refreshToken` | String   | Stores the latest refresh token. |
| `isVerified`   | Boolean  | Email verification status.       |
| `createdAt`    | Date     | Creation timestamp.              |
| `updatedAt`    | Date     | Last update timestamp.           |

---

# Conversation Collection

Represents a conversation created by a user.

Each conversation belongs to exactly one user.

## Fields

| Field        | Type     | Description                                   |
| ------------ | -------- | --------------------------------------------- |
| `_id`        | ObjectId | Conversation identifier.                      |
| `userId`     | ObjectId | Reference to the owner.                       |
| `title`      | String   | Conversation title.                           |
| `isPinned`   | Boolean  | Indicates whether the conversation is pinned. |
| `isArchived` | Boolean  | Archive status.                               |
| `createdAt`  | Date     | Creation timestamp.                           |
| `updatedAt`  | Date     | Last update timestamp.                        |

---

# Turn Collection

Each user prompt is stored as a separate turn.

A conversation can contain multiple turns.

## Fields

| Field            | Type     | Description                         |
| ---------------- | -------- | ----------------------------------- |
| `_id`            | ObjectId | Turn identifier.                    |
| `conversationId` | ObjectId | Parent conversation.                |
| `prompt`         | String   | User prompt.                        |
| `activeModels`   | Array    | AI models selected for this prompt. |
| `createdAt`      | Date     | Creation timestamp.                 |
| `updatedAt`      | Date     | Last update timestamp.              |

---

# Response Collection

Stores responses generated by AI providers.

One prompt can produce multiple responses.

Example:

Prompt

```
Explain JWT Authentication
```

Responses

- Gemini
- OpenAI
- Groq

Each response is stored as an independent document.

## Fields

| Field       | Type     | Description                          |
| ----------- | -------- | ------------------------------------ |
| `_id`       | ObjectId | Response identifier.                 |
| `turnId`    | ObjectId | Parent turn.                         |
| `provider`  | String   | AI provider.                         |
| `model`     | String   | AI model used.                       |
| `status`    | String   | Response status.                     |
| `content`   | String   | AI-generated text.                   |
| `error`     | String   | Error message (if generation fails). |
| `createdAt` | Date     | Creation timestamp.                  |
| `updatedAt` | Date     | Last update timestamp.               |

---

# Collection Relationships

## User → Conversation

Relationship

```text
One-to-Many
```

One user can own multiple conversations.

---

## Conversation → Turn

Relationship

```text
One-to-Many
```

Each conversation can contain multiple prompts.

---

## Turn → Response

Relationship

```text
One-to-Many
```

Each prompt can generate multiple AI responses.

# Indexing Strategy

The following fields should be indexed for faster queries.

| Collection   | Indexed Field    |
| ------------ | ---------------- |
| User         | `email`          |
| Conversation | `userId`         |
| Turn         | `conversationId` |
| Response     | `turnId`         |

These indexes improve lookup performance for authentication and conversation retrieval.

---

# Design Decisions

### Separate Conversation Metadata

Conversation details are stored separately from prompts and responses.

This keeps conversations lightweight and improves query performance when displaying the sidebar.

---

### Separate Turns

Each prompt is stored independently.

Benefits:

- Easier pagination
- Better scalability
- Cleaner history management

---

### Separate Responses

Responses are stored independently instead of embedding them inside a turn.

Benefits:

- Supports multiple AI providers
- Simplifies adding new providers
- Easier response tracking
- Better scalability

---
